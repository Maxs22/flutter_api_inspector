# Timeline Model Specification

## Purpose

The `flutter_api_inspector` timeline is the in-memory store of
API call records. It is a ring buffer of `ApiTraceRecord` objects
keyed by start time, ordered descending (newest first) when
rendered, and guarded by a fixed capacity so a long debug session
cannot exhaust memory. The model layer exposes
`ApiTraceRecord`, `ApiTraceRequest`, `ApiTraceResponse`,
`ApiTraceOutcome`, and the ring buffer itself. By default the
model is privacy-conscious: with the default
`ApiTraceConfig.details == {ApiTraceDetail.minimal}`, no body or
header fields are stored. All testable behavior in this file is
verifiable with `flutter test` unit tests against
`package:flutter_test`.

## Requirements

### Requirement: REQ-MODEL-001 — ApiTraceRecord schema

`ApiTraceRecord` MUST be an immutable value object with the
following fields: `id` (string, generated by the package),
`name` (string, developer-supplied label), `startedAt` and
`completedAt` (`DateTime`), `method` (string), `url` (`Uri`),
`statusCode` (nullable `int`), `duration` (`Duration`), `outcome`
(`ApiTraceOutcome`), `capturedDetails`
(`Set<ApiTraceDetail>`), `request` (nullable
`ApiTraceRequest`), `response` (nullable `ApiTraceResponse`),
`error` (nullable `Object`), and `extra`
(`Map<String, Object?>`).

**TDD contract**: `flutter test` MUST record a RED → GREEN →
TRIANGULATE → REFACTOR sequence in `apply-progress.md` for the
test(s) satisfying this requirement.

#### Scenario: Record exposes all required fields with correct types

- GIVEN a fully populated `ApiTraceRecord` constructed by the
  timeline
- WHEN each field is read
- THEN the field types match the schema (id: String, name:
  String, startedAt: DateTime, completedAt: DateTime, method:
  String, url: Uri, statusCode: int?, duration: Duration,
  outcome: ApiTraceOutcome, capturedDetails:
  Set<ApiTraceDetail>, request: ApiTraceRequest?, response:
  ApiTraceResponse?, error: Object?, extra: Map<String,
  Object?>)

#### Scenario: Record fields are immutable

- GIVEN a populated `ApiTraceRecord`
- WHEN the test code references the record's fields
- THEN the compile-time type system rejects any attempt to
  reassign a `final` field

### Requirement: REQ-MODEL-002 — ApiTraceOutcome enum

`ApiTraceOutcome` MUST be a three-state enum with the cases
`success`, `error`, and `cancelled`.

**TDD contract**: `flutter test` MUST record a RED → GREEN →
TRIANGULATE → REFACTOR sequence in `apply-progress.md` for the
test(s) satisfying this requirement.

#### Scenario: Enum has exactly three cases

- GIVEN the `ApiTraceOutcome` type
- WHEN the values are enumerated
- THEN the set equals `{success, error, cancelled}`

### Requirement: REQ-MODEL-003 — Ring buffer capacity is configurable

The timeline's ring buffer MUST be created with capacity
`ApiTraceConfig.timelineCapacity` (default 200 — locked answer
to open question #2). When a new record is added to a full
buffer, the oldest record MUST be evicted silently and the new
record appended at the head (preserving the newest-first
ordering invariant).

**TDD contract**: `flutter test` MUST record a RED → GREEN →
TRIANGULATE → REFACTOR sequence in `apply-progress.md` for the
test(s) satisfying this requirement.

#### Scenario: Default capacity holds exactly 200 records

- GIVEN a fresh timeline with `timelineCapacity = 200` (the
  default)
- WHEN 200 records are inserted
- THEN the timeline size is exactly 200
- AND the most recently inserted record is at the head

#### Scenario: Oldest record evicted when capacity is exceeded

- GIVEN a timeline with capacity 3 and three existing records
  named A, B, C
- WHEN a fourth record D is inserted
- THEN the timeline size is still 3
- AND the timeline order (head to tail) is D, C, B
- AND record A has been evicted silently (no error, no event)

#### Scenario: Capacity honored when configured explicitly

- GIVEN `ApiTraceConfig(timelineCapacity: 5)`
- WHEN 6 records are inserted
- THEN the timeline size is 5
- AND the oldest record is gone

### Requirement: REQ-MODEL-004 — Newest-first chronological ordering

The timeline MUST expose its records in descending order by
`startedAt` (newest first). Records inserted with the same
`startedAt` MUST be ordered by insertion order — the later
insertion wins the tie.

**TDD contract**: `flutter test` MUST record a RED → GREEN →
TRIANGULATE → REFACTOR sequence in `apply-progress.md` for the
test(s) satisfying this requirement.

#### Scenario: Newest record first

- GIVEN three records inserted with `startedAt` values
  T1 < T2 < T3
- WHEN the timeline is iterated in render order
- THEN the sequence is the record at T3, the record at T2, the
  record at T1

#### Scenario: Insertion order breaks tie on identical start time

- GIVEN two records inserted with the same `startedAt`
- WHEN the timeline is iterated in render order
- THEN the second-inserted record appears before the
  first-inserted record

### Requirement: REQ-MODEL-005 — Privacy-conscious default

When a call is captured with `capturedDetails ==
{ApiTraceDetail.minimal}` only, the resulting `ApiTraceRecord`
MUST NOT contain a request body, a response body, or request or
response headers. The `request` and `response` fields MUST be
either `null` or present as instances whose body and header maps
are empty (locked product decision: avoid leaking tokens, PII,
or large payloads by accident).

**TDD contract**: `flutter test` MUST record a RED → GREEN →
TRIANGULATE → REFACTOR sequence in `apply-progress.md` for the
test(s) satisfying this requirement.

#### Scenario: Minimal capture has no body or headers

- GIVEN a default `ApiTraceConfig` (details = {minimal}) and an
  `execute` callback that returns an `ApiTraceResponse` with a
  populated `responseBody`, `requestBody`, and full header maps
- WHEN a developer calls `ApiTrace.call(...)`
- THEN the record's `response.responseBody` is null
- AND the record's `response.responseHeaders` is empty
- AND the record's `request.requestBody` is null
- AND the record's `request.requestHeaders` is empty

#### Scenario: Headers-only capture includes headers but not bodies

- GIVEN a `detailOverride` of `{ApiTraceDetail.headers}` and an
  `execute` callback that returns an `ApiTraceResponse` with both
  headers and a body
- WHEN a developer calls `ApiTrace.call(...)`
- THEN the record's `response.responseHeaders` is non-empty
- AND the record's `response.responseBody` is null

### Requirement: REQ-MODEL-006 — Response body truncation

When `ApiTraceDetail.response` or `ApiTraceDetail.full` is in the
captured detail set, the response body stored in
`ApiTraceResponse.responseBody` MUST be truncated to
`ApiTraceConfig.maxResponseBodyBytes` (default 4 KB — locked
answer to open question #5). Bytes beyond the limit MUST be
dropped, with the resulting `responseBody` being a parsed
representation of the prefix.

**TDD contract**: `flutter test` MUST record a RED → GREEN →
TRIANGULATE → REFACTOR sequence in `apply-progress.md` for the
test(s) satisfying this requirement.

#### Scenario: Response body truncated to default 4 KB

- GIVEN `ApiTraceConfig.maxResponseBodyBytes = 4096` and an
  `execute` callback whose response body is a 10 KB string
- WHEN the call is captured at `ApiTraceDetail.response` level
- THEN the stored body length is at most 4096 bytes (or
  characters for text bodies)

#### Scenario: Response body truncation honors configured limit

- GIVEN `ApiTraceConfig(maxResponseBodyBytes: 128)` and an
  `execute` callback whose response body is 1024 bytes
- WHEN the call is captured at `ApiTraceDetail.response` level
- THEN the stored body length is at most 128 bytes

### Requirement: REQ-MODEL-007 — Reentrancy preserves records

The timeline ring buffer MUST be safe to use from a single Dart
isolate under reentrant async operations. Concurrent
`ApiTrace.call` invocations that interleave their `execute`
callbacks MUST each produce a record, and the timeline MUST
preserve the deterministic ordering described in
REQ-MODEL-004. (Dart is single-threaded per isolate, so
"thread-safety" in this spec means reentrancy, not multi-thread
synchronization.)

**TDD contract**: `flutter test` MUST record a RED → GREEN →
TRIANGULATE → REFACTOR sequence in `apply-progress.md` for the
test(s) satisfying this requirement.

#### Scenario: Two concurrent calls each produce a record

- GIVEN two `ApiTrace.call(...)` invocations launched without
  awaiting the first before starting the second
- WHEN both complete
- THEN the timeline size has grown by exactly two
- AND the two new records have distinct ids
- AND each record's `duration` is non-negative

### Requirement: REQ-MODEL-008 — No disk persistence in v1

The timeline ring buffer MUST be in-memory only. The package
MUST NOT write records to disk, shared preferences, SQLite, or
any other persistent store. Killing the app process MUST drop
the timeline (per the proposal's non-goals).

**TDD contract**: `flutter test` MUST record a RED → GREEN →
TRIANGULATE → REFACTOR sequence in `apply-progress.md` for the
test(s) satisfying this requirement. (The test asserts the
behavioral consequence: a fresh `ApiTrace` instance sees an
empty timeline. Dependency-graph absence of `dart:io` File
writes / `SharedPreferences` is a static property, but the
observable contract is "no record survives process restart".)

#### Scenario: Timeline resets across process restart

- GIVEN a test that records 5 calls and then constructs a fresh
  `ApiTrace` singleton (simulating a process restart)
- WHEN the new instance is queried
- THEN the timeline is empty (size 0)
- AND no record from the previous instance is observable

## Out of scope

- Disk persistence (SQLite, shared_preferences, file export).
- Encryption at rest.
- TTL-based eviction (only capacity-based eviction is in v1).
- Multi-window / multi-tab isolation (single timeline per app
  instance).
- Pagination of the timeline (the panel shows the whole ring).
- A query language for the timeline (only the UI filter chips
  are in v1).
- Telemetry / upload of any record.
- Direct editing of records (records are immutable).
- Multi-thread synchronization primitives (Dart is
  single-threaded per isolate; reentrancy is the relevant
  concern).
